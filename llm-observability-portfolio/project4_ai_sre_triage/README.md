# Project 4: AI SRE Triage Bot

## What this is

When a Datadog LLM Observability monitor fires, this bot:
1. Receives the webhook from Datadog
2. Fetches the failing spans from the Datadog API
3. Sends them to GPT-4o for root-cause analysis
4. Posts a structured, actionable hypothesis to Slack — within 30 seconds of the alert

This is a stripped-down implementation of the pattern behind Datadog Bits AI SRE,
built to understand how LLM-powered on-call tooling actually works.

## Architecture

```
Datadog Monitor fires
        │
        ▼ (webhook, JSON payload)
FastAPI /webhook/datadog
        │ (returns 200 immediately)
        ▼ (background task)
fetch_failing_spans()    ←── Datadog Spans API v2
        │
        ▼
analyze_with_llm()       ←── GPT-4o + structured prompt
        │
        ▼
post_to_slack()          ←── Block Kit notification
        │
        ▼
Slack message:
  🚨 AI SRE Triage — [RAG Bot] Faithfulness Alert
  Root cause: Prompt template v1.2 regression detected...
  Remediation: 1. Roll back to v1.1  2. ...
```

### Datadog webhook configuration

In Datadog: Integrations → Webhooks → New Webhook
- URL: `https://your-host:8000/webhook/datadog`
- Payload: `{"monitor_name": "$EVENT_TITLE", "text": "$TEXT_ONLY_MSG", "alert_type": "$ALERT_TYPE", "tags": $TAGS}`
- Enable "Send Authentication Header" with your WEBHOOK_SECRET

Apply the webhook to any LLM Observability monitor (from Projects 1, 2, or 3).

## Design decisions

### Background tasks over sync processing
Datadog expects a webhook response within 5 seconds. Span fetching + LLM
analysis takes 10-20 seconds. FastAPI `BackgroundTasks` solves this cleanly
without needing a full message queue for a PoC.

### Span count cap at 5 for LLM prompt
Raw span objects are large. Summarizing 5 spans into a compact text
representation keeps the prompt under 2000 tokens — fast and cheap.

### GPT-4o for analysis (not gpt-4o-mini)
On-call triage is high-stakes and low-volume. This is the right place to
spend on the premium model — a wrong root cause hypothesis is more expensive
than the $0.01 model cost.

## Interview talking points

- "The interesting engineering problem here is the 5-second webhook timeout.
  You can't do LLM analysis synchronously. I used FastAPI BackgroundTasks for
  the PoC, but in production you'd want a durable queue — SQS or Celery —
  so analysis survives a server restart."

- "The prompt is deliberately structured to produce JSON. That lets you route
  the confidence field — high-confidence analyses page the on-call engineer,
  low-confidence ones go to Slack for morning review. You're not just adding
  LLM analysis, you're adding triage filtering."

- "This connects back to LLM Observability fundamentally: the quality of the
  triage is only as good as the span data. If spans don't have model name,
  error message, and latency tagged, the LLM has nothing useful to analyze.
  It's a forcing function for good instrumentation."
